# Gmail 个人 AI 邮件分析系统 — 产品与技术框架(精简版)

> 一句话定位:只供你自己每天用的 AI 邮件分析助手——定时拉取多个 Gmail 新邮件,分级 + 摘要,每天给一份"今日要点"日报。不做客户端、不做多租户、不做模型网关、不做成本统计。

---

## 0. 结论先行

| 决策项 | 结论 |
|---|---|
| 邮箱读取 | **Gmail API(OAuth 只读)**,放弃 Playwright |
| 关键前提 | OAuth 应用发布为 **Production**(个人用<100、免验证),否则 refresh token 7 天失效 |
| 多 Gmail | 每个邮箱授权一次、各存一份 refresh token,服务循环刷新 |
| AI 调用 | **直接调用你已有的固定模型 API**,不引入 LiteLLM、不做多模型路由、不做成本统计 |
| 部署 | **Railway 单服务**(FastAPI 常驻 + Railway Cron + SQLite/PG) |
| 开源依赖 | 唯一硬依赖 = Gmail 官方客户端库;Inbox Zero 仅作分类/Prompt 写法的参考,不 fork |
| Token 优化 | 只做三条够用的(增量、规则预过滤、正文截断),不堆缓存/聚类/批处理框架 |

**一句话路线**:Railway 上一个 FastAPI 服务 + 一个 cron。Cron 定时 → Gmail 增量拉新邮件 → 规则粗筛丢噪声 → 正文清洗截断 → 调你固定的模型做分级+摘要 → 写库 → 每日聚合成日报推送。

---

## 1. 设计边界(明确不做)

- **不做**:多租户、用户体系、登录、会员、支付、复杂权限、高可用、微服务。
- **不做**:模型网关 / 多模型路由层 / Token 成本统计 / 用量看板。
- **不做(初期)**:自动回信、自动归档退订等写操作——放二期。
- **做**:多 Gmail 只读增量拉取 → 固定模型分级与摘要 → 每日日报。
- **原则**:一个服务 + 一个库 + 一条 cron。能直连就不抽象,能存库去重就不另起缓存层,能轮询就不上消息队列。

---

## 2. 邮箱读取:Gmail API(OAuth 只读)

### 2.1 为什么是它(一句话回顾)

Playwright 复用浏览器会话上不了云(机房 IP + headless 登录 Gmail 必被 Google 风控拦"This browser or app may not be secure")、易失效、维护重。Gmail API 的唯一代价是每个邮箱一次性授权,换来上云 + 长期稳定 + 低维护。结论已定,不再展开。

### 2.2 落地要点

| 项 | 做法 |
|---|---|
| 权限 | 只需 `gmail.readonly`(纯分析)。二期写回标签再加 `gmail.modify` |
| **发布状态** | **必须 Production**:Testing+External 下 refresh token 7 天失效;Production(免验证、个人用<100、首次点掉一次"未验证应用"警告)后长期有效 |
| 多账号 | 每个 Gmail 各跑一次授权流,各存一份 refresh token;服务启动循环刷新 |
| 增量同步 | 用 Gmail **History API**(存 `historyId`),只拉新邮件,绝不全量重扫 |
| 触达方式 | 个人用 **cron 定时轮询**即可,无需 Pub/Sub 推送 |
| token 兜底 | 捕获 `invalid_grant`(改密码/6 月未用/撤销触发)→ 标记该账号"需重新授权"→ 通知你重跑 |

## 3. 部署:Railway 单服务形态

| 组件 | 承载 | 说明 |
|---|---|---|
| GitHub | 代码 + CI | 推送即部署 |
| **Railway 服务** | 常驻 FastAPI | 提供拉取/分析逻辑;若要只读日报看板也挂这个服务 |
| **Railway Cron** | 定时任务 | 一条拉取(每 N 分钟)+ 一条日报(每天定时) |
| **存储** | SQLite(挂卷,最轻)或 Railway Postgres | 单用户优先 SQLite;要并发/远程查询再上 PG |
| **密钥** | 环境变量 | 各 Gmail 的 refresh token、你固定模型的 API key/base |

整套就一个 Railway 项目、一个服务、若干环境变量。组件越少越好维护,与你 RHCLOUD 的 FastAPI-on-Railway 同构。

## 4. 系统架构(精简)

### 4.1 数据流

```text
[Railway Cron 每N分钟]
   → 对每个 Gmail(refresh token) 调 History API 增量拉新邮件(只读)
   → 规则预过滤(List-Unsubscribe 头 / 发件人白黑名单 / 正则) ── 丢弃噪声邮件
   → 正文清洗 + 截断(HTML→文本,去引用历史/签名/页脚,按长度截断)
   → 调用固定模型(单次结构化调用,返回 分类 + 重要度 + 一句话 + 摘要)
   → 写入 DB(按 Gmail message id;已存在则跳过,天然不重复分析)
[Railway Cron 每天定时]
   → 按分类/发件人聚合当日结果 → 生成"今日要点"日报 → 推送(Markdown / 飞书卡片 / 邮件)
```

### 4.2 模块清单(已删去网关/缓存/成本模块)

| 模块 | 职责 | 实现 |
|---|---|---|
| Fetcher | 多账号增量拉取 | google-api-python-client + History API |
| Prefilter | 规则粗筛丢噪声 | 自研,纯函数 |
| Cleaner | HTML→文本、去噪、截断 | 自研小工具 |
| Analyzer | 调固定模型做分级+摘要 | 直连你的模型 API,一封一次结构化调用 |
| Store | 元数据/分类/摘要;按 message id 去重 | SQLite / PG |
| Digest | 日报聚合、渲染、推送 | 自研 + 复用 doc-page(Markdown/飞书) |
| Scheduler | 定时拉取/日报 | Railway Cron |

> 去重不需要独立缓存层:增量同步 + 按 Gmail message id 写库,本身就保证同一封邮件只分析一次。

## 5. AI 分析:直接调用固定模型

### 5.1 调用方式

- 直接用你已有的固定模型 API,**不引入 LiteLLM、不做网关、不做成本统计**。
- 默认**单模型**:配 `LLM_API_BASE` + `LLM_API_KEY` + `LLM_MODEL` 三个环境变量,所有分析任务都打这一个端点。
- 每封(过了预过滤的)邮件**一次结构化调用**,让模型一并返回:

```json
{
  "category": "紧急·需回复 | 金融·账户告警 | 法律·合同 | 重要通知 | 订阅·营销 | 社交其他",
  "importance": 1,
  "one_line": "一句话说这封邮件是什么",
  "summary": "重要度高时给要点(日期/金额/待办),否则留空"
}
```

### 5.2 可选:便宜/贵两档(非必须,不是路由框架)

如果你想"便宜模型分类、贵模型只摘要重要邮件":那只是再配一个 `SMART_MODEL` 端点 + 一个 `if importance >= 4` 分支,两个直连调用而已,不构成"模型路由模块"。
若你的两个固定模型请求格式不同(如 OpenAI 格式 vs Anthropic 格式),内置一个约 30 行的薄适配函数即可——仍远比 LiteLLM 轻。默认建议先单模型,够用再加这档。

### 5.3 个性化分类(贴你的实际场景,纯邮件分级,不含任何投资/法律建议)

| 分类 | 触发示意 | 处理 |
|---|---|---|
| 紧急·需回复 | 个人/工作直发、含截止日期或问句 | 置顶,二期自动预草拟 |
| 金融·账户告警 | 券商、保证金、对账单类发件人 | 高亮单列(仅识别提醒,不给操作建议) |
| 法律·合同 | 律所、法院、合同/传票类 | 高亮提醒尽快人工处理 |
| 重要通知 | 学校、政府、银行账户安全 | 摘要 + 要点 |
| 订阅·营销 | 含 `List-Unsubscribe` | 折叠,二期一键退订 |
| 社交其他 | 其余 | 一句话带过 |

## 6. Token:够用即可(不过度设计)

只做下面三条,都是近乎零成本、且本来就该做的:

1. **增量同步**:用 `historyId` 只处理新邮件,不全量重扫——这是省 Token 的根本。
2. **规则预过滤**:进模型前用 `List-Unsubscribe` 头 + 发件人白黑名单先丢掉营销/噪声邮件,只把"需思考"的送进模型——砍掉的就是最大的一块调用量。
3. **正文清洗 + 截断**:HTML 转纯文本、去引用历史/签名/页脚,按长度上限截断后再喂模型——单封 token 直接降下来。

不做缓存框架、不做近重复聚类、不做批处理、不做 Prompt 缓存调优。需要时再加,初版不上。

## 7. 功能与 MVP 分期

### 7.1 MVP(第一阶段,优先做)

1. 接入 1–3 个 Gmail(OAuth 只读,Production 发布)。
2. 增量拉取新邮件。
3. 固定模型分级 + 分类(用 §5.3 口径)+ 重要度。
4. 重要邮件给摘要 + 要点(日期/金额/待办)。
5. **每日"今日要点"日报**:一份聚合输出(Markdown / 飞书 / 邮件择一)。
6. 增量同步 + 规则预过滤(从第一天就有)。
7. 简单存储(SQLite/PG)。

> 验收:每天早上收到"X 封新邮件,其中 Y 封需关注",点开就是分级摘要——够每天用。

### 7.2 第二阶段(增强)

- 起草回复。
- 跨邮箱关键词/语义检索。
- 写回 Gmail 标签 / 自动折叠营销(`gmail.modify`)。
- 自定义规则、按发件人特殊处理。
- 多渠道推送(飞书 / Telegram)。
- 附件解析(PDF 对账单→抽取关键数字)。

## 8. 工作量与交给 AI 分工

### 8.1 工时

精简自建、单模型直连、无网关无成本模块:到"每天能用"约 **1–2 周**(AI 写 ~80%),维护成本低。

### 8.2 适合全权交给 AI(Coding Plan)写

Gmail API 封装(授权流/增量/解析)、规则预过滤、HTML→文本清洗与截断、模型直连调用 + JSON 解析、日报渲染(Markdown/飞书卡片)、SQLite 表结构与读写、Dockerfile/Railway 配置。这些边界清晰、文档完备,AI 出活快。

### 8.3 需要你把关(不全交 AI)

分类口径与重要度规则(你对"重要"的个人定义:券商/法律/学校等)、固定模型选哪个、密钥与 token 安全、在你真实邮件上迭代 Prompt 质量。

## 9. 风险兜底(各留一条即可)

| 风险 | 触发 | 兜底 |
|---|---|---|
| refresh token 失效 | 改密码 / 6 月未用 / 撤销 | 捕获 `invalid_grant` → 标记"需重新授权" → 通知你重跑 |
| Gmail API 限流 | 拉取过密 | 增量 + 退避重试 + 拉取间隔可调 |
| 模型偶发故障 | provider 侧 | 重试 + 失败该封跳过、下轮补 |

## 10. 落地步骤

1. 建 Google Cloud 项目,启用 Gmail API,配 OAuth(External),**发布为 Production**,逐个授权你常用的 Gmail,各存 refresh token。
2. 在 Railway 建一个 FastAPI 服务 + 一条拉取 cron + 一条日报 cron;存储先用挂卷 SQLite。
3. 配 §5.1 的固定模型三个环境变量,接 §4 架构,做增量 + 预过滤 + 截断三条。
4. 把 §8.2 清单丢给 Coding Plan 出初版代码,你只调分类口径与 Prompt。
5. 日报先用 Markdown,稳定后切飞书卡片(复用 doc-page)。

> 需要的话,下一步我可以展开:Gmail API 授权 + Production 发布完整步骤、单模型结构化调用的 Prompt 模板、或 Railway 服务 + cron + SQLite 的项目骨架(目录与环境变量清单)。
